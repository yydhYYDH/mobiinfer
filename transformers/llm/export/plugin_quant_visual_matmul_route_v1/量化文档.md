## 可行性分析

### 当前状态
- 6 个视觉分块已导出为 HiAI NPU `.om` 文件，位于 `plugin_quant_visual_matmul_route_v1/`。
- 量化校准输入目前由 `collect_visual_act_stats_v6.py` 通过 `generate_random_calibration_sample()` **随机生成**。
- 每个分块样本均为一个 `.npz` 文件，包含 3 个张量（验证了 `model_visual_plugin_matmul_calib_seq608` 中现有的格式）：
  - `hidden_states_in`: `[1, 608, 1024]` fp16
  - `rotary_pos_emb`: `[2, 608, 1, 64]` fp16 (堆叠的 `cos|sin`，rotary_dim=64 = head_dim 128 / 2)
  - `attention_mask`: `[1, 608, 608]` fp16 (全零)
- 文件名约定：`chunk_XX_sample_YYY.npz`，由 `load_calibration_samples()` 通过 `chunk_{chunk_index:02d}_sample_*.npz` glob 模式加载。

### MNN 运行时如何为每个分块提供这些输入
在 `omni.cpp::qwen2VisionProcess()` 中，视觉预模块 (`visual_pre_model`) 产生 `preOut = [hidden_states, rotary_pos_emb]`，`attention_mask` 则由独立构建。然后统一分块循环（约第 903 行）对每个分块 `i` 执行：
- OM 分块路径：将 `curHidden`, `preOut[1]` (rotary), `attention_mask` 转换为 float 向量。
- MNN 分块路径：`chunkIn = {curHidden, preOut[1], attention_mask}`。

关键点在于 **每个分块的 `hidden_states_in` 是前一个分块的输出**（第一个分块除外，它来自预模块）。`rotary_pos_emb` 和 `attention_mask` 在所有分块中是**共享的**，且与图像无关（仅 `seq_len` 相关）。

因此，一个包含真实图像的前向传播过程会**为每个分块产生一个有效的 (hidden, rotary, mask) 三元组**，这正是 npz 校准格式所需的内容。

### 核心可行性问题：MNN VARP 的形状能否与 npz 格式匹配？

| 输入 | MNN VARP 形状 | npz 形状 | 结论 |
|---|---|---|---|
| `hidden_states_in` | `[1, S, 1024]` (来自预模块/前一个分块) | `[1, S, 1024]` | ✅ 完全匹配 |
| `rotary_pos_emb` | `preOut[1]` — 预模块输出，打包为 cos\|sin | `[2, S, 1, 64]` | ⚠️ 必须验证 MNN 精确的打包方式；适配器中的 `_normalize_rotary` 接受 rank-5 或 rank-4 (cos/sin 作为 dim 0)，因此只要 MNN 预模块输出 `[2,S,1,rotary_dim]`，打包方式即匹配 |
| `attention_mask` | 非窗口分支中的 `[1, S, S]` 全零 (第 772-773 行) | `[1, S, S]` 全零 | ✅ 匹配 |

### 关键可行性风险

1. **rotary 打包不匹配**：这是首要风险。`_ExportAttentionAdapter._normalize_rotary` 要求 rank-4 的情况为 `[2, S, 1, D]`，其中 `[0]`=cos，`[1]`=sin。MNN 预模块必须以该精确布局输出 rotary。鉴于现有的导出流程生成完全相同的 ONNX 分块，且校准器随机生成器模仿了 `[2,S,1,64]`，MNN 预模块输出几乎确定是兼容的 —— 但这必须通过在检查点 dump 并用 `_normalize_rotary` 测试来验证。建议：同时 dump MNN 预模块的 rotary 输出，并与导出时的 `visual.rotary(position_ids)` 进行对比。

2. **seq_len 匹配**：npz 使用 `seq_len=608`。通过真实图像生成的 `seq_len = grid_t*grid_h*grid_w`，其中 grid 基于 `mVisionHeight/mVisionWidth`（图像尺寸，按对齐大小取整）。要获得 `S=608`，图像必须产生精确的 608 个 patch。**608 = 32 × 19 = 16 × 38...** —— 你需要确定精确的 (H, W) 组合，使得 `grid_h*grid_w = 608`（`grid_t=1`，`merge_size=2`）。这是一个真实的限制，但可解：要么选择产生 608 的图像尺寸，要么重新生成 `seq_len` 与你实际输入图像匹配的校准输入（第二种方法更好 —— 让数据驱动 seq_len）。

3. **没有 Python npz 写入器可用**：MNN C++ 引擎没有 npz/npy 写入器。C++ 端将 dump 原始 float32/float16 二进制文件（通过 `_Input`/`writeMap` 或直接 ofstream）；Python 后处理脚本会将这些 `.bin` + `.json`（形状）转换为 `.npz`。这是最干净且侵入性最小的路径，并符合记忆中 `[[feedback_mobiinfra_oh_deps]]` 的指导：新逻辑保留在 MNN 工具/引擎中，不会给手机端应用带来新的依赖。

4. **Deepstack 输出**：某些分块（`deepstack_visual_indexes=[5,11,17]`）产生额外的 deepstack 输出。仅校准分块 *输入* 需要 deepstack，因此这只影响下一个分块的 `hidden_states_in` —— `curHidden` 流水线已经处理了这一点。无需进行 deepstack 特定的 dump 即可捕获输入。

5. **单次前向 → 多个图像**：每张图像产生一组（每个分块一个）三元组。要获得每个分块 N 个校准样本（当前为 `num_samples=4`），需要 N 张不同的图像。每张图像也会馈送所有 6 个分块，因此 N 张图像 → 每个分块 N 个样本，全部来自一次批处理运行。

### 推荐架构（侵入性最小）

添加一个 **仅用于 dump 的代码路径**，由 `config.json` 中的标志（例如 `visual_dump_chunk_inputs`）控制，并在 `mChunkUseOm` 为 false 时在统一分块循环中激活（即分块作为 MNN 模块运行，这是 x86/CPU 主机上的情况）。对于每个分块 `i`，在调用 `onForward` 之前，将三个输入 VARP 以原始 fp32 二进制 + sidecar JSON 写入。然后使用一个小型 Python 脚本转换为 npz。

这可以在不对手机端应用进行修改的情况下复用（标志默认关闭；手机端使用 OM 分块，从不运行此路径），并且根据记忆约束，手机端的 `libMNN.so` 不受影响。

```cpp
static bool dumpVarpToBinFile(const std::string& path, VARP var,
                              /*out*/ std::vector<int>& shape) {
    if (var == nullptr) return false;
    auto info = var->getInfo();
    if (info == nullptr) return false;
    VARP readVar = (info->type.code==halide_type_float && info->type.bits==32)
                   ? var : _Cast(var, halide_type_of<float>());
    auto readInfo = readVar->getInfo();
    auto ptr = readVar->readMap<float>();
    if (ptr == nullptr) return false;
    std::ofstream ofs(path, std::ios::binary);
    ofs.write(reinterpret_cast<const char*>(ptr),
              readInfo->size * sizeof(float));
    shape.assign(readInfo->dim.begin(), readInfo->dim.end());
    return true;
}
```

## 分析结论:量化机制 = PyTorch(透明) + 华为 dopt 黑盒(不透明)

这个目录的量化**不是用 PyTorch 自带量化工具**(`torch.quantization` / PyTorch 官方 PTQ),也不是 MNN 自带的 quant tool。它用的是**华为 DDK 的 dopt(do_opt)工具链**,以 PyTorch 模块为载体做伪量化(伪量化/QAT-style fake quant)。

### 1. 量化工具来源:华为 dopt,不是 PyTorch 自带

主脚本 `visual_plugin_quant_matmul_route.py:59-65` 导入的量化函数全部来自:

```python
from dopt.dopt_lm.do_opt import (
    generate_config_file, generate_quant_params,
    optimize_model, set_calibrate_state, set_quant_state,
)
```

`dopt` 来自 `DDK-tools-next-6.0.1.0/tools/tools_dopt/dopt_pytorch_py3`,**整个工具链都是编译好的 `.so`**(C 扩展),没有可读的 Python 源码:

```
dopt/dopt_lm/do_opt.so      ← optimize_model / generate_quant_params 等的真实实现
dopt/dopt_lm/builder.so
dopt/notrain_pytorch/pytorch_quant_tool.so
dopt/strategy/strategy_manager.so
dopt/dopt.so ... (共 20+ 个 .so)
```

所以 `optimize_model`、`generate_quant_params` 内部如何算 scale/offset、如何分组量化、`Quant_aigc_ptq` 策略具体做什么——**全部不透明,是华为闭源实现**。

### 2. PyTorch 在其中的角色(透明部分)

PyTorch 只负责**承载模型和跑前向**,本身不做量化决策。具体分工:

| 阶段 | 谁执行 | 透明度 |
|------|--------|--------|
| 加载模型、构造 chunk module | `LlmModel.from_pretrained`(MNN 导出端的 PyTorch 模型) | ✅ 透明 |
| `optimize_model(chunk, cfg)` | **dopt.so** | ❌ 黑盒 |
| `set_calibrate_state(qmodel, True)` | **dopt.so**(给 Linear 挂上 `quant_op` hook) | ❌ 黑盒(但 hook 注入结果可见,见下) |
| `qmodel(sample...)` 跑前向 | **PyTorch**(`torch.no_grad` 普通前向) | ✅ 透明 |
| `set_calibrate_state(qmodel, False)` | **dopt.so**(结束统计、固化 scale) | ❌ 黑盒 |
| `torch.save(qmodel.state_dict())` | PyTorch | ✅ 透明 |
| `generate_quant_params(qmodel, ...)` | **dopt.so**(生成 OMG 用的 quant_params_file) | ❌ 黑盒 |
| 导出 ONNX | PyTorch `torch.onnx.export` + 本脚本后处理 | ✅ 透明 |
| OMC 编译 | 华为 `omg` 工具(`run_visual_plugin_matmul_omc.sh`) | ❌ 黑盒 |

关键:**校准就是用 PyTorch 跑前向**(line 956-964),但跑的不是原模型——是 dopt 用 `optimize_model` 改造过的模型,每个 Linear 被注入了 `quant_op`(伪量化算子),前向时会**统计激活 min/max 并做 fake quant**。这部分统计逻辑在 dopt 的 `.so` 里,不透明。

### 3. 不透明中的"可观测窗口"

虽然算法黑盒,但有两处产物是可读的,能反推 dopt 做了什么:

**(a) `dopt_config.chunk_00.json`(透明 JSON)**——量化配置,你能完全控制:

```json
"blocks.0.mlp.linear_fc1": {
  "quant_strategy": "Quant_aigc_ptq",   // ← 策略名(实现黑盒)
  "weight": { "bit": 8, "group_size": 128, "weight_algo": "min_max" },
  "input": { "bit": 16, "input_algo": "min_max", "unsigned_quant": true }
}
```

- `quant_strategy`:`Quant_aigc_ptq`(AIGC 模型专用 PTQ)或 `Quant_act_weight_eco`,**具体策略实现黑盒**
- `weight_algo: min_max`、`input_algo: min_max`:标了是 min-max,但实际是否纯 min-max 还是带平滑/EMA 由 dopt 决定
- `qkv_proj` 被设成 `"float"`(不量化),单独的 q/k/v/o/fc1/fc2 才量化

**(b) `calibrated_chunk_00.pth`(透明 state_dict)**——dopt 注入的量化参数可见:

每个被量化的 Linear 多出 `quant_op` 子模块,含 6 类参数:

```
blocks.0.mlp.linear_fc1.quant_op.weight_quantizer.max   (4096,1)  ← 逐通道权重 max
blocks.0.mlp.linear_fc1.quant_op.weight_quantizer.min   (4096,1)  ← 逐通道权重 min
blocks.0.mlp.linear_fc1.quant_op.weight_quantizer.s     (4096,1)  ← scale(算出来的)
blocks.0.mlp.linear_fc1.quant_op.weight_quantizer.offset(4096,1)  ← zero point
blocks.0.mlp.linear_fc1.quant_op.weight_quantizer.bit   ()        ← bit=8
blocks.0.mlp.linear_fc1.quant_op.input_quantizer.s      ()        ← 激活 scale(标量,逐层非逐通道)
blocks.0.mlp.linear_fc1.quant_op.input_quantizer.max/min()        ← 校准统计的激活范围
```

从中能反推:
- **权重是 per-channel 量化**(`s/max/min` shape 是 `[out,1]`,即每个输出通道一个 scale)
- **激活是 per-tensor 量化**(`input_quantizer.s` 是标量 `()`)
- **group_size=128** 但 state_dict 里权重 scale 是 per-channel,说明分组量化(group-wise)的 scale 应该在 `quant_params_file` 里体现(由 `generate_quant_params` 生成,二进制不可读)
- scale `s` 由 dopt 根据 min/max + bit + algo 算出,**具体公式(min_max 对称/非对称、是否 clip)**黑盒

**(c) `quant_params_file`(二进制,不透明)**——3MB,OMG 工具消费的最终量化参数,格式是华为私有二进制,不可读。

### 4. 完整数据流与透明度标注

```
PyTorch chunk module (透明)
   │  optimize_model(chunk, dopt_config)   ← dopt.so 黑盒:注入 quant_op
   ▼
qmodel (PyTorch module + quant_op hooks)
   │  set_calibrate_state(True)            ← dopt.so 黑盒:开启激活统计
   │  for sample in calib_inputs:          ← PyTorch 透明前向(但 hook 统计黑盒)
   │      qmodel(hidden, rotary, mask)
   │  set_calibrate_state(False)           ← dopt.so 黑盒:固化 scale
   ▼
calibrated_chunk.pth (透明 state_dict: weight + quant_op.{s,offset,max,min,bit})
   │  generate_quant_params(qmodel, ...)   ← dopt.so 黑盒:转成 OMG 格式
   ▼
quant_params_file (二进制,不透明) + fake_quant_weight.pth
   │  torch.onnx.export(fake_quant weights) ← PyTorch 透明导出
   ▼
visual_blocks_npu_0.onnx (透明,可 netron 查看)
   │  omg --compress_conf quant_params_file ← 华为 omg 黑盒编译
   ▼
visual_plugin_matmul_quantized.om (NPU 离线模型,不透明)
```

### 5. 总结

- **不是 PyTorch 自带量化,也不是 MNN 的 quant tool**:是华为 DDK 的 **dopt(do_opt)** 工具链,核心全是 `.so` 闭源。
- **PyTorch 只做两件透明的事**:① 承载模型 ② 跑前向(校准样本)。量化决策、伪量化算子、scale 计算、参数导出全在 dopt。
- **不透明点**:`Quant_aigc_ptq` / `Quant_act_weight_eco` 策略具体算法、`min_max` 算子是否带平滑、激活统计是否带 EMA/移动平均、group-wise scale 如何切分、`quant_params_file` 二进制格式——全部在 `.so` 里看不到。
- **可观测/可控点**:`dopt_config.chunk_XX.json`(量化位宽/分组/算法名/unsigned 等可调)、`calibrated_chunk_XX.pth`(注入的 scale/offset/max/min 可读,可反推 per-channel 权重 / per-tensor 激活)、ONNX(netron 可查)。
- **校准输入的影响点**:因为校准是 dopt hook 在前向时统计激活 min/max,所以**真实校准输入直接决定 `input_quantizer.max/min` → 决定激活 scale `s`**(state_dict 里可见)。这就是为什么换真实图像后,激活量化范围会比随机输入更贴合真实分布——这是整个链路里唯一你能直接影响量化质量的环节,其余算法都封在 dopt 里。

### 作用 2:导出前把 Linear 改写成 plain MatMul 形态(`_FLinearMatmul`)

**文件**:`visual_plugin_quant_matmul_route.py:339-376`
**作用**:导出 ONNX 阶段,把 `nn.Linear` 替换成一个"权重 + bias 拆开、前向用 `F.linear`"的等价模块。目的是让 ONNX 里主权重层落成 **plain `MatMul + Add`**(而不是 `Gemm`),这样 OMG 编译器才能识别成 `MatMuls16s4Gen` 走 NPU 量化算子。这是**导出形态控制**,不涉及量化计算。

```python
# line 339-376
class _FLinearMatmul(torch.nn.Module):
    def __init__(self, m: torch.nn.Linear, algin_dim=1, squeeze_batch=False, unsqueeze_before_bias=False):
        super().__init__()
        self.in_features = m.in_features
        self.out_features = m.out_features
        ...
        # 对齐维度后,把原始 Linear 的 weight/bias 拷过来(只读 Parameter)
        self.weight = torch.nn.Parameter(m.weight.data.clone(), requires_grad=False)
        self.bias = (torch.nn.Parameter(m.bias.data.clone(), requires_grad=False)
                     if m.bias is not None else None)

    def forward(self, x):
        if self.squeeze_batch:
            x = x.reshape(x.shape[1], x.shape[2])
        out = torch.nn.functional.linear(x, self.weight)   # ← 落成 MatMul 的关键
        if self.bias is not None:
            out = out + self.bias                            # ← bias 单独 Add
        ...
        return out
```

```python

    for name, child in list(module.named_children()):
        if type(child) == torch.nn.Linear:
            setattr(module, name, _FLinearMatmul(child, ...))
        else:
            _replace_linear_with_flinear(child, ...)
```

**意义**:决定 ONNX 结构(`MatMul=32, Gemm=0`),是"导出长相"控制,不是量化本身。

## `qmodel` 是什么

`qmodel = optimize_model(chunk, cfg_path)`。

一句话:**`qmodel` 是 dopt 把你的原始 PyTorch chunk 模型改造后的"伪量化模型"** —— 一个仍然能像普通 `nn.Module` 一样跑 `forward()` 的 PyTorch 模块,但每个被量化的 `nn.Linear` 身上都被 dopt 注入了一个叫 `quant_op` 的子模块,里面挂了权重量化器和激活量化器。它**不是你原始的 chunk,也不是一个独立的量化引擎**,而是"原 chunk + 每层挂一个量化插件"的组合体。

### 1. 它怎么来的

```python
chunk, meta = load_visual_chunk(...)          # 原始 PyTorch 模块(作用1,纯浮点 Linear)
qmodel = optimize_model(chunk, cfg_path)      # ★ dopt.so 改造 chunk → qmodel
```

`optimize_model`(在 `dopt/dopt_lm/do_opt.so` 里,黑盒)做的事:
- 遍历 `chunk` 里所有 `nn.Linear`,对照 `dopt_config.chunk_00.json` 的 `layer_strategy`
- 对每个要量化的 Linear(如 `blocks.0.self_attn.q_proj`),在它身上**挂一个 `quant_op` 子模块**
- 不量化的(如 `qkv_proj` 被设成 `"float"`)不挂,保持原样

所以 `qmodel` 的结构 = 原 chunk 结构 + 每层多一个 `quant_op`。

### 2. 它内部长什么样(从 state_dict 反推)

`qmodel` 是 PyTorch 模块,`qmodel.state_dict()` 可存。看一个 Linear(`blocks.0.self_attn.q_proj`)名下的全部参数,就能看清 `qmodel` 里这一层长什么样:

```
blocks.0.self_attn.q_proj.
    ├── weight                          (1024,1024) float32   ← 原始权重(还在)
    ├── bias                            (1024,)     float32   ← 原始 bias
    └── quant_op.                       ← dopt 注入的量化插件
        ├── weight_quantizer.           ← 权重量化器(per-channel)
        │   ├── bit        () int64   = 8
        │   ├── max        (1024,1)         ← 每通道权重最大值
        │   ├── min        (1024,1)         ← 每通道权重最小值
        │   ├── s         (1024,1)         ← scale(算出的,per-channel)
        │   ├── offset    (1024,1)         ← zero-point(per-channel)
        │   └── unsigned_quant () = 0.0    ← 权重有符号量化
        └── input_quantizer.            ← 激活量化器(per-tensor)
            ├── bit        () int64   = 16
            ├── max        ()  = 9.46       ← 校准统计的激活最大值 ★ 真实输入影响这里
            ├── min        ()  = -11.21     ← 校准统计的激活最小值 ★ 真实输入影响这里
            ├── s          ()  = 3.15e-4   ← 激活 scale(算出的,per-tensor)
            ├── offset      ()  = 35535.0   ← zero-point
            └── unsigned_quant () = 1.0    ← 激活无符号量化
```

对照 `dopt_config` 的配置,一一对应:
- `weight.bit=8` → `weight_quantizer.bit=8`
- `weight_algo=min_max`、`group_size=128` → 决定 `s/offset` 怎么算
- `input.bit=16`、`input_algo=min_max`、`unsigned_quant=true` → `input_quantizer.bit=16`、`unsigned_quant=1.0`
- 权重 per-channel(`s` 是 `[1024,1]`),激活 per-tensor(`s` 是标量 `()`)

### 3. 它跑 `forward()` 时在干什么(校准阶段)

```python
set_quant_state(qmodel, weight_state=True, input_state=True)  # 打开权重/激活量化
set_calibrate_state(qmodel, True)                             # 打开"校准态"
with torch.no_grad():
    for sample in samples:
        _ = qmodel(hidden, rotary, mask)   # ← 前向
set_calibrate_state(qmodel, False)                            # 关闭校准态,固化 scale
```

`qmodel(sample...)` 表面上是普通 PyTorch 前向,但因为每层挂了 `quant_op`,实际流程是:

1. 数据进 `blocks.0.self_attn.q_proj`
2. `quant_op.input_quantizer` 的 **forward pre-hook** 先拦住输入,做两件事:
   - **校准态 ON**:统计这次输入的 min/max,累加进 `input_quantizer.min/max`(这就是激活范围统计)
   - (量化态 ON 时还会对输入做 fake quant,但校准阶段主要目的是统计)
3. 原始 `weight` 被 `weight_quantizer` 做 fake quant(用已算的 `weight_quantizer.s/offset`)
4. `F.linear(x_quantized, weight_quantized)` 算出本层输出
5. 输出进下一层,重复

**关键**:`input_quantizer.min/max` 在每个 sample 前向时被 dopt 的 hook 更新。所以:
- 你喂的校准样本不同 → 前向激活值不同 → `input_quantizer.min/max` 不同 → 算出的 `input_quantizer.s`(激活 scale)不同
- **这就是真实校准输入影响量化的落点**,而且这个变化在 state_dict 里完全可见

上面那组数值里 `input_quantizer.max=9.46, min=-11.21` 就是用你 dump 的真实图像校准出来的。如果是随机输入,这俩数会是 ±0.05 量级。

### 4. 校准完之后 `qmodel` 怎么用

```python

torch.save(qmodel.state_dict(), state_path)            # ← 存成 calibrated_chunk_00.pth
                                                      #   (含原 weight + 全部 quant_op 参数)

generate_quant_params(qmodel, ...)                     # ★ dopt 读 qmodel 里的 quant_op 参数,
                                                      #   生成 OMG 用的二进制 quant_params_file
```

- `calibrated_chunk_00.pth`:就是 `qmodel` 在校准完成那一刻的**完整 state_dict**(原权重 + 每层 `quant_op` 的 scale/offset/min/max/bit)。202MB,因为带了原 fp32 权重。
- `generate_quant_params`:dopt 从 `qmodel` 的 `quant_op` 里抽出 scale/offset,转成 OMG 私有二进制格式(`quant_params_file`,3MB)。

### 5. `qmodel` vs 原 `chunk` 的关系

| | 原 `chunk` | `qmodel` |
|---|---|---|
| 来源 | `load_visual_chunk`(MNN 导出端加载) | `optimize_model(chunk, cfg)`(dopt 改造) |
| 结构 | 纯 `nn.Linear` | 原 Linear + 每层挂 `quant_op` |
| forward | 浮点前向 | 浮点前向 + 每层 fake quant + 激活统计 |
| state_dict | 只有 weight/bias | weight/bias + 每层 `quant_op.{s,offset,max,min,bit,...}` |
| 谁改的 | PyTorch | **dopt.so**(黑盒) |
| 作用 | 承载模型 | 校准时的伪量化模型 |

### 6. 为什么叫"伪量化"

`qmodel` 里的 `quant_op` 做的是 **fake quant**(伪量化):前向时用 float 模拟量化误差(把权重/激活 round 到量化网格再转回 float),模型权重本身仍是 float32 存的。真正的低比特整数权重,是后面 `generate_quant_params` → OMG 编译阶段才生成的。所以 `qmodel` 的作用是**在浮点域里估算量化参数和量化误差**,而不是直接产出 INT8/INT4 权重。

## 总结

`qmodel` = `optimize_model(chunk, cfg_path)` = **dopt 给原 chunk 每个目标 Linear 挂上 `quant_op` 量化插件后的伪量化 PyTorch 模型**。它:
- 仍是 PyTorch `nn.Module`,能正常 `forward()`
- 每层多了 `quant_op.{weight_quantizer, input_quantizer}`,里面是 dopt 算的 scale/offset/min/max
- 校准时前向触发 `input_quantizer` 的 hook 统计激活范围(真实输入影响这里)

## `qmodel` 详细代码解释

`qmodel = optimize_model(chunk, cfg_path)`(在 `calibrate_and_export_quant` 里,line 949)。

### 一、`qmodel` 的诞生:do_opt.so 对原 chunk 做的"原地改造"

```python
# visual_plugin_quant_matmul_route.py:948-949
chunk, meta = load_visual_chunk(...)              # 原始 chunk:纯 nn.Linear 的 PyTorch 模块
qmodel = optimize_model(chunk, cfg_path)          # dopt 改造 → qmodel
```

`optimize_model` 来自 `dopt.dopt_lm.do_opt`(编译成 `do_opt.so`,黑盒)。但通过运行时反射,可以精确描述它做了什么:

1. **遍历 chunk 里所有 `nn.Linear`**,对照 `dopt_config.chunk_00.json` 的 `layer_strategy`
2. **逐层决定策略**(运行日志可见):
   ```
   blocks.0.self_attn.q_proj   using strategy: Quant_aigc_ptq   ← 要量化
   blocks.0.self_attn.qkv_proj using strategy: float            ← 不量化,保持原样
   ```
3. **对每个要量化的 Linear,把 `nn.Linear` 替换成 `dopt_lm.modules.qlinear.QLinear`**;不量化的保留 `nn.Linear`

反射证据(`optimize` 前后 `q_proj` 类型对比):
```
optimize 前:  q_proj -> torch.nn.modules.linear.Linear
optimize 后:  q_proj -> dopt_lm.modules.qlinear.QLinear    ← dopt 替换了
              qkv_proj -> torch.nn.modules.linear.Linear   ← 策略 float,没换
```

所以 **`qmodel` 还是原来的 `VisualBlocksChunkNoDS` 容器**(类型没变),只是里面被量化的 `nn.Linear` 子模块被换成了 `QLinear`。`qmodel` ≠ 一个新对象,而是 **"被 dopt 动过手脚的原 chunk"**。

### 二、`qmodel` 内部:三层结构

每个被量化的层(以 `blocks.0.self_attn.q_proj` 为例)变成:

```
q_proj : QLinear                         (dopt_lm.modules.qlinear.QLinear)
├── weight        (1024,1024) Parameter  ← 原始权重,原样保留
├── bias          (1024,)     Parameter  ← 原始 bias
└── quant_op : QClass                    (dopt_lm.quantizer.qmanager.QClass)  ← dopt 注入的"量化管理器"
    ├── weight_quantizer : MinMaxGrid    (dopt_lm.quantizer.qmin_max.MinMaxGrid)  ← 权重量化器
    └── input_quantizer  : MinMaxGrid    (dopt_lm.quantizer.qmin_max.MinMaxGrid)  ← 激活量化器
```

对应 state_dict 里的 key(这就是 `calibrated_chunk_00.pth` 里能看到的):
```
blocks.0.self_attn.q_proj.weight                          ← 原权重
blocks.0.self_attn.q_proj.bias                            ← 原 bias
blocks.0.self_attn.q_proj.quant_op.weight_quantizer.{bit,max,min,s,offset,unsigned_quant}
blocks.0.self_attn.q_proj.quant_op.input_quantizer.{bit,max,min,s,offset,unsigned_quant}
```

三个 dopt 类的职责(从反射到的签名推断):

| 类 | 文件(.so) | 关键方法 | 职责 |
|---|---|---|---|
| `QLinear` | `dopt_lm/modules/qlinear.so` | `forward(self, x)`、`set_run_mode`、`set_inplace_quant` | 包装原 Linear,在前向里调用 `quant_op` 做伪量化 |
| `QClass` | `dopt_lm/quantizer/qmanager.so` | `forward(self, weight, x)`、`set_quant_state(weight,input)`、`set_calibrate_state(state)` | 量化管理器:协调 weight/input 两个量化器,提供状态开关 |
| `MinMaxGrid` | `dopt_lm/quantizer/qmin_max.so` | `forward(self, x)`、`get_min_max_value(x)`、`get_clip_range`、`get_scale_shape`、`init_param(bit)` | 真正的 min-max 量化器:统计范围、算 scale/offset、做 fake quant |

### 三、`qmodel` 前向时发生了什么(校准的核心)

校准代码(line 951-964):
```python
set_quant_state(qmodel, weight_state=True, input_state=True)   # ① 打开量化
set_calibrate_state(qmodel, True)                              # ② 打开校准态(开始统计)
with torch.no_grad():
    for sample in samples:
        reset_kv_cache(qmodel)
        _ = qmodel(hidden, rotary, mask)                       # ③ 前向
set_calibrate_state(qmodel, False)                             # ④ 关校准态(固化 scale)
```

每次 `qmodel(sample)` 前向,数据流经某个 `QLinear`(比如 `q_proj`),内部流程(由反射到的签名推断):

1. **`QLinear.forward(x)`** 被调用
2. 它调用 **`quant_op.forward(weight, x)`**(`QClass.forward(self, weight, x)`)
3. `QClass` 分别调用两个 `MinMaxGrid`:
   - `weight_quantizer`:对 `weight` 做伪量化
     - 用已存的 `weight_quantizer.s/offset`(per-channel)把权重 round 到量化网格再转回 float
   - `input_quantizer`:对输入 `x` 处理,**这里分两种状态**:
     - **校准态 ON**(`set_calibrate_state(True)`):`get_min_max_value(x)` 统计本次输入的 min/max,更新 `input_quantizer.min/max` 缓冲;`init_param`/算 `s` 和 `offset`
     - **校准态 OFF**:`get_clip_range` + 用 `s/offset` 对 x 做 fake quant
4. 用伪量化后的 weight 和 x 算 `F.linear`,输出进下一层

**关键结论**:
- **激活统计是 `MinMaxGrid.get_min_max_value` 在 `input_quantizer` 里做的**,不是 PyTorch 的 forward pre-hook(反射确认 `QLinear._forward_pre_hooks` 为空,量化逻辑直接写在 `QLinear.forward` 内部)
- **你喂的校准样本 → `qmodel` 前向 → `input_quantizer.get_min_max_value` 统计 → 更新 `input_quantizer.min/max` → 算出 `input_quantizer.s`**(激活 scale)
- 这就是真实校准输入影响量化的精确落点。前面看到的 `input_quantizer.max=9.46, min=-11.21, s=3.15e-4` 就是真实图像跑出来的

### 四、`MinMaxGrid` 的量化参数含义

反射到 `MinMaxGrid.__init__(self, father, bit, per_channel=True, unsigned_quant=False, quant_id=0, keep_neg_max=True, **argv)`,对照 state_dict 和 config:

| 参数 | weight_quantizer | input_quantizer | 来源 |
|---|---|---|---|
| `bit` | 8 | 16 | `dopt_config` 的 `weight.bit`/`input.bit` |
| `per_channel` | True(`s` shape `[1024,1]`) | False(`s` shape `()`) | dopt 内部:权重逐通道,激活逐层 |
| `unsigned_quant` | 0.0(有符号) | 1.0(无符号) | `dopt_config` 的 `unsigned_quant` |
| `keep_neg_max` | True(默认) | True | dopt 内部默认 |
| `s`(scale) | `[1024,1]` per-channel | `()` per-tensor | `get_scale_shape` + min/max 算出 |
| `offset`(zero-point) | `[1024,1]` | `()` | 同上 |
| `min`/`max` | `[1024,1]` 权重范围 | `()` 激活范围(校准统计) | `get_min_max_value` |

`get_scale_shape(per_channel, father)` 这个方法名直接说明:**per-channel 时 scale 形状对齐权重输出维度(`[out,1]`),per-tensor 时是标量**。
